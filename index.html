Storage Account Types

Azure offers different storage account types optimized for various scenarios and workloads.

Standard General-Purpose v2

Best for: Most scenarios

  • Blobs, Files, Queues, Tables
  • All redundancy options
  • All access tiers
  • Cost-effective for most workloads

Premium Block Blobs

Best for: High transaction rates

  • Block blobs and append blobs only
  • Low latency operations
  • LRS and ZRS redundancy
  • Analytics and IoT scenarios

Premium File Shares

Best for: Enterprise applications

  • Azure Files only
  • High performance file shares
  • SMB and NFS protocols
  • Database and development tools

Premium Page Blobs

Best for: Virtual machine disks

  • Page blobs only
  • Optimized for VHDs
  • Low latency random I/O
  • Azure VM disk storage
Recommendation: For most use cases, Standard General-Purpose v2 provides the best balance of features, performance, and cost.

Storage Service Types

Azure Storage provides four main data services, each designed for specific data storage needs.

1. Azure Blob Storage

Purpose: Object storage for unstructured data

  • Block Blobs: Text and binary data, ideal for files and documents
  • Append Blobs: Optimized for append operations, great for logging
  • Page Blobs: Random access files up to 8 TB, used for VHDs

Use Cases: Images, videos, backups, data lakes, streaming data

2. Azure Files

Purpose: Managed file shares in the cloud

  • SMB and NFS protocol support
  • Can be mounted on Windows, Linux, and macOS
  • Lift-and-shift applications
  • Shared application settings and diagnostic logs

Use Cases: Replace on-premises file servers, shared application data

3. Azure Queue Storage

Purpose: Message storage for asynchronous processing

  • Store millions of messages
  • Messages up to 64 KB in size
  • Accessed via HTTP/HTTPS
  • Decoupling application components

Use Cases: Workflow processing, task queues, service communication

4. Azure Table Storage

Purpose: NoSQL key-value store

  • Schema-less design
  • Structured non-relational data
  • Automatic scaling
  • Fast and cost-effective

Use Cases: User data, device information, metadata storage

Redundancy Features

Azure Storage provides multiple redundancy options to protect your data against planned and unplanned events.

Redundancy Type Copies Protection Level Use Case
LRS
Locally Redundant Storage		 3 copies in single datacenter Protects against server rack and drive failures Non-critical data, cost-sensitive scenarios
ZRS
Zone-Redundant Storage		 3 copies across availability zones Protects against datacenter-level failures High availability requirements within a region
GRS
Geo-Redundant Storage		 6 copies (3 local + 3 in paired region) Protects against regional outages Disaster recovery scenarios
GZRS
Geo-Zone-Redundant Storage		 6 copies across zones and regions Maximum durability and availability Critical data requiring highest protection
Read Access: Add "-RA" prefix (RA-GRS, RA-GZRS) to enable read access to the secondary region for even higher availability.

Durability Comparison

  • LRS: 99.999999999% (11 nines) durability
  • ZRS: 99.9999999999% (12 nines) durability
  • GRS/RA-GRS: 99.99999999999999% (16 nines) durability
  • GZRS/RA-GZRS: 99.99999999999999% (16 nines) durability

Access Tiers

Azure Blob Storage offers different access tiers to optimize costs based on data access patterns.

Hot Tier

Optimized for: Frequently accessed data

  • Highest storage costs
  • Lowest access costs
  • Active datasets
  • Data for processing

Example: Images on a website, active application data

Cool Tier

Optimized for: Infrequently accessed data

  • Lower storage costs than Hot
  • Higher access costs than Hot
  • Minimum 30-day retention
  • Short-term backup

Example: Recent backups, data accessed occasionally

Cold Tier

Optimized for: Rarely accessed data

  • Lower storage costs than Cool
  • Higher access costs than Cool
  • Minimum 90-day retention
  • Long-term storage

Example: Older backups, compliance data

Archive Tier

Optimized for: Rarely accessed long-term data

  • Lowest storage costs
  • Highest access costs
  • Minimum 180-day retention
  • Rehydration required (hours)

Example: Legal archives, historical records
